# Архитектурные заметки и компромиссы (notes.md)

В данном документе фиксируются ключевые предположения, компромиссы (tradeoffs) и архитектурные решения, принятые при разработке прототипа **AI Brief Decoder Lite**.

## Принятые допущения (Assumptions)

- **Локальный PostgreSQL:** Предполагается наличие запущенного PostgreSQL инстанса (через docker-compose) для локальной разработки.
- **Однопользовательский режим:** На этапе прототипа нет авторизации пользователей, все запуски принадлежат единому пространству.

## Технологические компромиссы (Tradeoffs)

- **Использование PydanticAI:** Выбран фреймворк PydanticAI. Он полностью совместим и согласован со схемами Pydantic и экосистемой FastAPI. Другие альтернативы (например, LangChain) избыточны, слишком тяжеловесны и сложны для легковесного прототипа. Более того, Pydantic является повсеместным стандартом в Python-разработке, что гарантирует простоту понимания кода любым новым инженером в команде.
- **Абстракция провайдеров vs Усложнение кода:** Создание интерфейса `BaseLLMProvider` усложняет код на начальном этапе, но является критически важным для тестируемости проекта (Fake-провайдер) и гибкости смены LLM.
- **WXT vs Raw Extension:** Использование WXT ускоряет сборку и обеспечивает современный DX (React HMR), но добавляет зависимость от стороннего фреймворка.

## Дополнительные архитектурные решения

- **Конфигурация по окружениям (Dynaconf):**
  - `[default]` — общие настройки для всех сред.
  - `[local]` — локальные переопределения для разработки.
  - `[testing]` — настройки под тесты (например, имя базы `brief_decoder_test`).
  - `[production]` — публичные настройки для прода (например, реальный провайдер `LLM_PROVIDER = "gemini"` вместо `fake`).
- **Где лежат секреты (пароли, ключи API):**
  - **Локально:** в файле `.secrets.toml` (он скрыт и добавлен в `.gitignore`).
  - **На проде / в GitLab CI:** передаются напрямую через системные переменные окружения ОС (прокидываем в k8s/Helm или Docker env). Наш код в первую очередь ищет их в системе напрямую (без префиксов типа `APP_`), игнорируя файлы.
  - Переменная `ENV_FOR_DYNACONF=production` прописывается в Dockerfile бэкенда или в манифестах k8s для включения продакшен-слоя.
- **Автоматическая подготовка тестовой БД:** Для упрощения локального запуска и интеграции с CI-раннерами в `conftest.py` встроена логика `setup_test_db`. Она сама проверяет наличие базы `brief_decoder_test` в СУБД, создает её при необходимости и накатывает/сбрасывает таблицы.
- **Контракт Liveness Probe для k8s:** Спецификация эндпоинта `/api/health` переведена на схему `HealthCheckState` с алиасом `Status: Ok` (через Pydantic V2 `ConfigDict`), гарантируя точное соблюдение требований Kubernetes Liveness Probes.
- **Чистая архитектура и инверсия зависимостей (Repository Pattern):**
  - Реализован паттерн Repository с поддержкой асинхронного контекстного менеджера (`BriefRepository` Protocol + `SQLAlchemyBriefRepository` реализация).
  - Вся работа с базой данных (включая транзакции, `commit()` и `rollback()`) скрыта внутри контекстного менеджера репозитория. Сервисы (`BriefService`) больше не вызывают методы коммита напрямую.
  - Бизнес-логика в `BriefService` оптимизирована с точки зрения производительности: получение текста брифа и запись результатов разбора производятся в двух независимых коротких транзакциях, а долгий внешний сетевой вызов LLM API выполняется **вне открытой сессии БД**, предотвращая исчерпание пула соединений (Connection Pool Starvation).
  - Вся работа с обновлениями переведена с нетипизированных словарей (`**kwargs`) на строгую Pydantic схему `BriefUpdate` и переименована под единую сущность `Brief` (убрана избыточная терминология `decode` и `run`).
  - Введены кастомные исключения в `exceptions.py` (`BriefNotFoundError`, `RepositoryContextError`) для явной обработки ошибок на границах слоев.
- **Динамический фейковый провайдер:** Для исключения хардкода в `FakeLLMProvider` и обеспечения реалистичного тестирования локально была интегрирована библиотека `polyfactory`. Ответы генерируются Pydantic-фабриками с подмешиванием случайных наборов требований и рисков. Инициализация seed-генератора от MD5-хэша входного текста гарантирует воспроизводимость результатов при повторной отправке одного и того же брифа.
- **Отказоустойчивость сети и паттерн «Декоратор»:**
  - Логика повторных попыток (retries) и таймаутов полностью вынесена из слоя бизнес-логики (`BriefService`) во внешний декоратор `RetryingLLMProvider`. Это сохраняет сервис «чистым» от низкоуровневых инфраструктурных проблем.
  - Настройки `LLM_MAX_ATTEMPTS` и `LLM_TIMEOUT` полностью параметризованы через Dynaconf.
- **Мониторинг длительности задач:**
  - В схему данных добавлены поля `started_at` и `finished_at`, что позволяет отслеживать точное время, затраченное ИИ-моделью на разбор каждого брифа.
- **Восстановление после сбоев (Zombie Task Cleanup):**
  - Разработан фоновый процесс `CleanupService` для автоматической очистки задач, зависших в статусе `processing` дольше заданного лимита (`CLEANUP_TIMEOUT_MINUTES` в настройках).
  - Воркер интегрирован в жизненный цикл FastAPI (`lifespan`) как асинхронная задача `asyncio.create_task`, гарантируя корректное завершение (`cancel()`) при выключении сервера.
- **Безопасные ошибки и интеграция с Sentry:**
  - Создан словарь перевода технических ошибок `SAFE_ERRORS` на русский язык для безопасного отображения пользователям.
  - Подключен Sentry SDK для перехвата критических ошибок во время работы фоновых задач.
